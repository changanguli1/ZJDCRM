# ZJDCRM 产业园区招商线索管理系统设计规格

## 1. 目标

ZJDCRM 是部署在 `cfzzs.custard.top` 的产业园区招商线索管理系统，用于 10–30 人招商团队替代 Excel 台账。系统覆盖线索获取、企业与联系人维护、跟进时间线、招商阶段流转、空间匹配、签约落地、提醒、报表、导入、导出审批、权限和审计。

本规格以用户提供的《产业园区招商线索管理系统 PRD V1.0》为产品需求基准。PRD 中 V1.0 MVP、业务规则、非功能要求和验收标准属于本次交付范围；V1.1、V2.0 只预留扩展接口，不在首版实现完整业务。

## 2. 技术架构

采用 Cloudflare 原生单体架构：

- 前端：React、TypeScript、Vite、React Router、TanStack Query、React Hook Form、Zod、ECharts。
- UI：基于可访问性组件构建桌面优先的中文管理界面，适配 Chrome、Edge 最新两个主版本及 1366×768 以上分辨率。
- API：Cloudflare Pages Functions，按模块拆分路由与服务。
- 数据库：Cloudflare D1，使用版本化 SQL migration。
- 附件与导出文件：Cloudflare R2。
- 会话与限流状态：D1 持久化；短时请求限制可使用进程内快速拒绝加 D1 最终状态。
- 定时任务：独立 Cloudflare Worker Cron，处理跟进提醒、过期导出和历史文件清理。
- 部署：GitHub 公开仓库连接 Cloudflare Pages；生产分支部署到 `cfzzs.custard.top`。

选择该架构的原因：

- 用户规模与数据量适合 D1 和 Pages Functions，不需要额外维护 Java、MySQL、Redis 和云服务器。
- 前端、API、数据库、附件和域名均由 Cloudflare 管理，发布链路短。
- 系统保持模块化单体，符合 PRD 的 MVP 架构要求，并保留后续拆分通知、导出任务和 AI 服务的边界。

## 3. 页面与导航

### 3.1 登录

- `/login`：内部用户登录。
- 用户名支持内部账号、手机号或邮箱。
- 连续 5 次失败锁定账号 30 分钟。
- 登录成功后按角色返回菜单与权限。
- 登录成功、失败、退出和锁定均写入审计日志。

### 3.2 普通业务端

- `/`：招商首页看板。
- `/clues`：招商线索列表。
- `/clues/new`：新增招商线索。
- `/clues/:id`：招商线索详情。
- `/clues/:id/edit`：编辑招商线索。
- `/unassigned`：未分配线索池。
- `/spaces`：空间资源列表。
- `/spaces/:id`：空间详情。
- `/reminders`：站内提醒。
- `/reports`：招商报表。
- `/imports`：有权限用户的数据导入。
- `/exports`：导出申请和历史。
- `/profile`：个人资料与修改密码。

普通菜单不显示系统管理入口。前端隐藏不作为安全边界，所有 API 仍执行后端权限校验。

### 3.3 管理端

`/admin` 为管理端入口，不出现在普通用户菜单中。只有具备管理权限的账号可访问。

管理端包括：

- 系统状态。
- 员工、部门与账号管理。
- 角色、菜单、按钮和数据范围权限。
- 招商阶段、行业、渠道、标签、跟进方式、流失原因、空间状态等字典。
- 园区、楼宇、楼层和空间资源维护。
- 导入任务、失败明细和模板。
- 导出申请审批、文件状态和下载审计。
- 登录日志、操作日志和权限变更日志。
- 品牌名称、Logo、登录页文案、系统公告、看板模块开关等网站设置。
- 备份执行记录和恢复说明。

初始管理员账号为 `admin`。用户已在会话中提供初始密码，该密码通过 Cloudflare Secret 注入，只以 PBKDF2 哈希保存在数据库中，不写入公开仓库、构建产物、设计文档或日志。管理员可在后台修改密码。

## 4. 角色和权限

系统采用 RBAC、数据范围和记录级权限的组合模型。

### 4.1 默认角色

- 超级管理员：全部数据与系统配置。
- 管理层：全部招商数据、看板、报表和导出审批。
- 招商主管：本团队数据、线索分配、负责人转移和团队报表。
- 招商人员：本人负责的数据、新增线索、联系人、跟进和空间匹配。
- 运营 / 综合岗：授权范围内的数据维护、空间维护、初始化导入、报表和导出申请。

### 4.2 权限组成

- 菜单权限：决定导航可见性。
- 动作权限：新增、编辑、阶段变更、分配、删除、恢复、导入、导出、审批等。
- 数据范围：本人、本团队、全部、指定部门。
- 记录级权限：负责人、被授权协作人和管理员。

列表、详情、统计、附件和导出均使用同一数据权限服务，避免只在列表层过滤。

## 5. 核心数据模型

招商线索是唯一主业务对象，不额外引入传统 CRM 的“商机”对象。

### 5.1 组织和权限

- `users`：账号、密码哈希、状态、部门、锁定状态。
- `departments`：部门树。
- `roles`：角色。
- `permissions`：菜单与动作权限。
- `user_roles`、`role_permissions`、`role_data_scopes`：授权关系。
- `sessions`：会话哈希、过期时间、IP 和 User-Agent。

### 5.2 招商业务

- `companies`：企业客户主数据。
- `clues`：招商线索主表。
- `contacts`：联系人。
- `clue_contacts`：线索与联系人关联。
- `followups`：跟进记录。
- `stage_histories`：阶段变更历史。
- `tags`、`clue_tags`：标签。
- `clue_collaborators`：协作人及读写权限。
- `attachments`：R2 文件元数据及业务关联。

### 5.3 空间资源

- `parks`：园区。
- `buildings`：楼宇。
- `floors`：楼层。
- `spaces`：空间、面积和状态。
- `clue_space_matches`：多个备选空间。
- `clue_landings`：实际落位空间、面积、日期和财源完成情况。

一个空间可被多个线索意向；意向不锁定空间。已签约或已落地后记录实际占用，不覆盖历史匹配。

### 5.4 流程与系统

- `notifications`：站内提醒。
- `import_jobs`、`import_job_rows`：导入任务与逐行结果。
- `export_requests`：导出申请、审批状态和理由。
- `export_files`：R2 导出文件、过期时间和下载状态。
- `audit_logs`：关键操作审计。
- `login_logs`：登录审计。
- `dictionaries`、`dictionary_items`：基础配置。
- `system_settings`：品牌和系统设置。
- `backup_records`：备份记录。

所有业务表包含 `created_at`、`created_by`、`updated_at`、`updated_by`。需要恢复的表增加 `deleted_at` 和 `deleted_by`。

## 6. 核心业务规则

### 6.1 查重

- 企业名称在写入前进行去空格、统一全半角、统一大小写和常见公司后缀处理。
- 企业名称标准化后相同，禁止直接新增。
- 联系人手机号相同，判定为重复。
- 导入中的重复数据记录为失败行，不静默覆盖。
- 管理员可在后续版本实现合并；V1.0 提供跳转现有记录和人工处理。

### 6.2 阶段流转

阶段默认顺序：

1. 新线索。
2. 已建档。
3. 初步接触。
4. 需求确认。
5. 重点跟进。
6. 考察洽谈。
7. 意向确认。
8. 签约推进。
9. 已签约。
10. 已注册 / 已落地。
11. 暂缓 / 流失。

规则：

- 阶段允许前进和后退。
- 每次变化必须填写原因，并在同一事务中写入阶段历史和时间线。
- 进入初步接触必须至少有一个联系人。
- 进入暂缓 / 流失必须填写原因。
- 进入已注册 / 已落地必须填写实际空间、实际面积、实际落地日期和财源完成情况。
- 只有负责人、主管或获授权人员可变更阶段。

### 6.3 分配和转移

- 手工新增时录入人默认是负责人。
- 导入且未指定负责人时进入未分配线索池。
- 主管可分配本团队成员。
- 负责人转移写入审计日志并通知新负责人。

### 6.4 删除和恢复

- 普通招商人员不能删除线索。
- 删除使用软删除。
- 已签约、已落地线索不能删除。
- 被空间、导出或其他关键记录引用的数据不能直接删除。
- 管理员可恢复软删除数据。

## 7. 跟进、提醒与附件

跟进记录包含：

- 跟进方式。
- 跟进时间。
- 跟进内容。
- 客户反馈。
- 核心卡点。
- 下一步动作。
- 下次跟进时间。
- 可选阶段变化。
- 附件。

下次跟进时间生成站内提醒。重点线索阶段变化通知主管；分配通知新负责人；导出申请通知管理层；审批结果通知申请人；标记流失通知主管。

附件上传执行：

- MIME 和扩展名双重校验。
- 单文件大小限制。
- 随机对象键，不信任原始文件名。
- 下载前执行对象权限校验。
- 删除业务记录时不立即物理删除 R2 对象，由清理任务延迟处理。

## 8. 看板与报表

所有指标支持时间范围筛选，并可点击进入对应权限范围内的明细。

### 8.1 首页

- 新增招商线索数。
- 阶段分布。
- 招商漏斗与转化率。
- 渠道来源。
- 人员负责项目数和跟进次数。
- 重点项目。
- 预计落位面积。
- 已签约和已落地数量。
- 预计产值和税收。
- 财源完成情况。
- 空间状态和面积。
- 待办提醒。

### 8.2 报表

- 招商漏斗。
- 客户来源分析。
- 招商业绩分析。
- 跟进效率分析。
- 空间面积和状态。
- 落地成果。

统计 API 复用数据权限查询条件。转化率明确使用阶段达到数除以前一阶段达到数，并处理分母为零。

## 9. Excel 导入

V1.0 提供固定 `.xlsx` 模板：

- 招商线索。
- 企业客户和联系人。
- 空间资源。

导入流程：

1. 上传文件到临时 R2 路径。
2. 校验模板版本、表头、类型、必填字段和字典值。
3. 执行企业名称与手机号查重。
4. 逐批写入 D1。
5. 展示成功数、失败数和逐行失败原因。
6. 记录导入人、时间、IP、文件摘要和结果。

1000 行以内导入目标在 30 秒内完成。为控制 Pages Functions 执行时间，任务分批提交，前端轮询进度。

## 10. 导出审批

普通招商人员没有导出权限。

导出流程：

1. 有申请权限的用户选择对象、字段、筛选范围并填写原因。
2. 系统保存申请时固化筛选条件和申请人的数据权限快照。
3. 管理层批准或驳回；驳回必须填写原因。
4. 批准后系统按当前有效权限和申请范围重新校验并生成文件。
5. 文件存储于私有 R2，下载令牌仅保存哈希。
6. 下载链接 24 小时有效。
7. 下载前再次验证用户身份、申请人与文件关系及有效期。
8. 申请、审批、生成和下载均写入审计日志。

## 11. 安全设计

- 密码使用 Web Crypto PBKDF2-SHA-256，随机盐和足够迭代次数；数据库不保存明文。
- 会话令牌使用密码学安全随机值，客户端使用 `HttpOnly`、`Secure`、`SameSite=Lax` Cookie；数据库只保存令牌哈希。
- 所有修改请求验证同源和 CSRF 令牌。
- API 使用统一认证、授权和数据范围中间件。
- 登录错误统一返回“账号或密码错误”，避免账号枚举。
- 登录失败按账号和 IP 记录并限流。
- 输入使用 Zod 校验；SQL 仅使用参数绑定。
- 响应设置 CSP、HSTS、`X-Content-Type-Options`、`Referrer-Policy` 和合理的 `Permissions-Policy`。
- 文件上传执行类型、大小和权限校验。
- 关键日志包含操作者、时间、IP、User-Agent、对象、动作和变更摘要，不记录密码、会话令牌或附件正文。
- 生产 Secrets 仅通过 Cloudflare Secret 或平台配置注入。

## 12. 错误处理

API 返回统一结构：

```json
{
  "ok": false,
  "error": {
    "code": "CLUE_STAGE_REQUIRE_CONTACT",
    "message": "进入初步接触阶段前至少需要一个联系人",
    "requestId": "..."
  }
}
```

- 400：请求格式或业务校验失败。
- 401：未登录或会话过期。
- 403：没有动作或数据权限。
- 404：对象不存在或无权发现对象。
- 409：重复、状态冲突或并发更新冲突。
- 413：附件或导入文件过大。
- 429：登录或接口请求过于频繁。
- 500：未处理异常，返回请求编号，不暴露堆栈。

关键更新使用 D1 事务或原子批处理。线索更新使用版本号进行乐观并发控制，避免覆盖其他用户刚保存的修改。

## 13. 性能与索引

关键索引：

- 线索：标准化企业名、负责人、阶段、渠道、创建时间、更新时间、删除状态。
- 联系人：手机号。
- 跟进：线索 ID、跟进时间、下次跟进时间。
- 空间：园区、楼宇、楼层、状态。
- 通知：接收人、已读状态、触发时间。
- 审计：操作者、对象、动作、时间。

列表全部后端分页。首页并行请求聚合指标，常用指标使用高效 SQL；必要时增加每日汇总表。目标：

- 3000 条线索下列表不超过 2 秒。
- 常用筛选不超过 2 秒。
- 首页常规查询不超过 3 秒。
- 支持 30 人以内同时在线。

## 14. 测试与验收

### 14.1 自动测试

- 单元测试：名称规范化、权限判断、阶段校验、转化率、下载有效期。
- 数据库测试：迁移、索引、软删除、唯一约束和事务。
- API 集成测试：认证、本人/团队/全部数据范围、越权访问、导入、导出审批。
- 组件测试：复杂表单、阶段变更、空间匹配和审批页面。
- Playwright 端到端测试：登录、线索闭环、主管分配、落地、导出审批、管理员配置。
- 安全测试：错误密码锁定、伪造角色、IDOR、CSRF、文件类型绕过、过期下载链接。

### 14.2 验收映射

PRD 第 16 章的账号权限、招商线索、联系人、跟进、空间、导入、导出和看板验收项全部转换为自动化测试或上线检查项。

## 15. 部署与运维

- GitHub 仓库公开，使用 MIT License。
- 仓库不包含 `.dev.vars`、Cloudflare Token、管理员明文密码、生产数据库或用户附件。
- GitHub 默认分支连接 Cloudflare Pages 生产环境。
- Pages 项目绑定 D1、R2 和生产 Secrets。
- 自定义域名配置为 `cfzzs.custard.top`，启用 HTTPS。
- D1 每日导出备份，R2 启用版本或按日清单；备份至少保留 30 天。
- 上线前创建默认部门、角色、字典、招商阶段和管理员账号。
- 发布后执行远程登录、权限、线索、附件、导入、导出、审计和移动宽度基本检查。

## 16. 项目边界

V1.0 不实现：

- 完整合同管理。
- 回款、租金和财务核算。
- 通用客服工单。
- 企业微信或飞书深度集成。
- AI 总结和推荐。
- 自动空间推荐。
- 完整移动端。
- 客户公海自动回收。
- 通用复杂审批流。

代码模型预留合同、回款、房间级空间、外部通知和 AI 模块的扩展位置，但不为尚未使用的功能提前增加复杂业务。
